# TechBrain 产品需求与版本规划

> 文档状态：初稿  
> 当前阶段：需求设计与基础架构规划  
> 产品定位：面向程序员的 AI 技术知识大脑平台

## 1. 项目概述

TechBrain 是一个面向程序员的个人技术知识资产管理平台。

系统用于统一管理个人在学习、工作和项目过程中积累的技术文档、学习笔记、实践经验及 AI 生成内容，并将这些零散信息沉淀为可持续增长、可追溯、可检索、可关联和可复用的个人知识库。

TechBrain 不是传统博客或单纯的笔记工具。它关注的是知识资产的长期积累与再利用，最终帮助开发者构建属于自己的“第二大脑”。

## 2. 背景与问题

程序员的技术知识通常分散在本地文件、博客平台、收藏夹、聊天记录、项目文档和各类笔记工具中，长期使用后容易出现以下问题：

- 知识入口分散，缺少统一管理方式。
- 文档格式和组织方式不一致，迁移与维护成本高。
- 依赖目录或记忆查找内容，检索效率低。
- 知识之间缺少关联，难以形成完整的技术体系。
- AI 生成内容停留在临时对话中，无法持续沉淀和复用。
- 内容不断增加后，重复、过时和互相矛盾的信息难以及时发现。

## 3. 产品目标

TechBrain 的核心目标是建立一套由开发者自主掌控、持续演进的个人技术知识体系。

### 3.1 近期目标

- 建立统一、稳定的 Markdown 知识存储规范。
- 实现知识的自动同步、结构化管理和全文检索。
- 降低知识记录、整理和再次查找的成本。
- 为后续 RAG、知识图谱和 AI 应用提供可靠的数据基础。

### 3.2 长期目标

- 建设个人技术百科。
- 建设个人技术搜索引擎。
- 建设个人 RAG 知识库。
- 建设个人技术知识图谱。
- 建设能够理解个人知识体系的 AI 技术助手。

### 3.3 非目标

现阶段不以以下能力为主要目标：

- 面向公众的博客发布、流量运营和内容社区。
- 多人实时协作文档编辑。
- 替代 Git 或云盘承担通用文件存储。
- 在早期版本中构建复杂的组织级权限体系。
- 让数据库或搜索引擎取代 Markdown 成为知识事实来源。

## 4. 目标用户与使用场景

### 4.1 目标用户

首要用户是需要长期积累技术知识的软件开发者，包括：

- 持续学习新技术的程序员。
- 需要整理项目经验和故障案例的工程师。
- 使用 AI 辅助学习和开发，希望沉淀高价值结果的用户。
- 希望建立个人技术体系与学习路线的用户。

产品早期以单用户、个人部署场景为主，并为未来扩展保留空间。

### 4.2 核心使用场景

1. 用户新增或修改本地 Markdown 文档，系统自动识别并同步。
2. 用户按分类、标签或更新时间浏览技术知识。
3. 用户通过关键词快速查找文档标题、正文、代码和元数据。
4. 用户收藏高频使用的知识，并查看最近更新内容。
5. 用户基于个人知识库提问，获得带原文引用的回答。
6. 用户查看某项技术与概念、工具、实践之间的关联关系。
7. 用户输入学习目标，由 AI 生成基于现有知识的学习路线和补充建议。

## 5. 核心设计原则

### 5.1 Markdown 是唯一事实来源

原始知识统一以 Markdown 文件保存。用户应当能够脱离 TechBrain 直接读取、编辑、迁移和通过 Git 管理这些文件。

MySQL、ElasticSearch 和 Milvus 均属于可重建的派生存储，不得成为无法从 Markdown 恢复的唯一数据来源。

### 5.2 可追溯

结构化数据、搜索结果、向量片段和 AI 回答都应能够追溯到对应的源文档及具体位置。

### 5.3 渐进式增强

系统先解决知识沉淀和检索，再逐步增加 AI 问答、知识图谱和智能学习能力。后续版本不能破坏基础知识管理能力。

### 5.4 本地优先与数据可控

用户应当掌控原始知识文件，并能够备份、迁移和重建派生数据。涉及外部模型时，应明确哪些内容会被发送到外部服务。

### 5.5 自动化但不失控

同步、切片、标签建议、关联发现等流程可以自动完成，但影响源知识的修改应由用户确认，避免 AI 静默改写事实来源。

## 6. 总体数据流

```text
Markdown 知识库（SSOT）
        │
        ▼
Knowledge Sync Service
        │
        ├──► MySQL：文档、分类、标签、关系等结构化数据
        │
        ├──► ElasticSearch：标题、正文、代码和元数据全文索引
        │
        └──► 文档切片与 Embedding
                    │
                    ▼
                 Milvus：向量索引
                    │
                    ▼
              RAG / AI 应用
```

同步服务是 Markdown 与各派生存储之间的边界，负责解析、校验、幂等同步、删除处理和状态记录。

## 7. 知识组织模型

### 7.1 文档

文档是知识管理的基本单元，对应一个 Markdown 文件。建议包含以下信息：

- 唯一标识。
- 标题与摘要。
- 文件路径。
- 分类和标签。
- 创建时间与更新时间。
- 文档状态。
- 内容哈希或版本标识。
- 来源及外部参考链接。

### 7.2 分类

分类用于表达相对稳定的知识归属，宜形成简单、清晰的层级结构，例如：

```text
后端开发
├── Python
├── MySQL
├── Redis
└── 消息队列

基础设施
├── Docker
└── Kubernetes

人工智能
├── AI
└── RAG
```

一篇文档原则上拥有一个主分类，避免多重目录造成归属混乱。

### 7.3 标签

标签用于表达跨分类的主题和特征，一篇文档可以包含多个标签，例如：

```text
ORM、性能优化、微服务、缓存、向量检索、最佳实践
```

系统应支持标签规范化，减少同义、大小写和单复数差异导致的重复标签。

### 7.4 知识关联

知识关联用于描述文档或概念之间的语义关系。关系至少应包含：

- 源节点。
- 目标节点。
- 关系类型。
- 关系说明。
- 创建来源：人工或 AI 建议。
- 可信状态：待确认或已确认。

可选关系类型包括：

- `属于`
- `依赖`
- `实现`
- `替代`
- `对比`
- `前置知识`
- `相关`

示例：

```text
SQLAlchemy ──包含能力──► joinedload
joinedload ──对比──► contains_eager
```

### 7.5 文档元数据建议

Markdown 可通过 Front Matter 描述元数据：

```yaml
---
id: sqlalchemy-joinedload
title: SQLAlchemy joinedload 使用指南
category: Python/SQLAlchemy
tags:
  - ORM
  - 性能优化
status: published
created_at: 2026-06-22
updated_at: 2026-06-22
---
```

具体字段、必填规则和兼容策略应在实现同步服务前形成独立规范。

## 8. 功能需求

### 8.1 V1.0：知识管理平台

#### 版本目标

完成个人知识库的基础建设，形成统一、可浏览、可维护、可检索的技术知识库。

#### 8.1.1 Markdown 知识同步

- 支持扫描指定知识库目录。
- 支持新增、修改、移动和删除文档的识别。
- 支持解析 Markdown 正文与 Front Matter。
- 同步过程应幂等，重复执行不会产生重复数据。
- 单篇文档失败不应阻塞整个同步任务。
- 记录同步时间、结果、失败原因和文档状态。
- 删除源文档后，应按既定策略清理或软删除派生数据。
- 支持全量同步，并为后续增量或事件驱动同步预留能力。

#### 8.1.2 分类管理

- 查看分类列表及层级。
- 新建、编辑、排序和删除分类。
- 查看分类下的文档。
- 分类删除前校验关联文档，并提供迁移方案。
- Markdown 中的分类信息与平台数据保持一致。

#### 8.1.3 标签管理

- 查看、新建、编辑、合并和删除标签。
- 按标签筛选文档。
- 展示标签使用次数。
- 识别并处理重复或近似标签。

#### 8.1.4 知识浏览

- 按分类、标签和更新时间浏览文档。
- 展示文档标题、摘要、分类、标签和更新时间。
- 支持分页或连续加载。
- 支持空状态、加载状态和同步异常提示。

#### 8.1.5 Markdown 渲染

- 正确渲染标题、列表、表格、引用、链接和图片。
- 支持代码块语法高亮。
- 支持目录导航与标题锚点。
- 对不可信 HTML 进行安全处理。
- 保留复制代码等面向开发者的易用能力。

#### 8.1.6 全文检索

- 使用 ElasticSearch 索引标题、正文、代码块、分类和标签。
- 支持关键词搜索与相关度排序。
- 支持分类、标签等条件过滤。
- 搜索结果显示命中摘要及关键词高亮。
- 搜索结果可直接定位到源文档。
- 索引可从 Markdown 和结构化数据中全量重建。

#### 8.1.7 收藏

- 收藏和取消收藏文档。
- 查看收藏列表。
- 收藏状态不修改 Markdown 正文，可作为用户行为数据单独保存。

#### 8.1.8 最近更新

- 展示最近新增和最近修改的文档。
- 支持按更新时间排序。
- 区分源文档更新时间与系统同步时间。

#### V1.0 验收标准

- 新增、修改或删除 Markdown 后，派生数据能够正确同步。
- 用户可通过分类和标签找到目标文档。
- 常用 Markdown 内容可正常、安全地渲染。
- 用户可通过关键词搜索正文和代码，并看到高亮结果。
- 用户可收藏文档并查看最近更新。
- MySQL 或 ElasticSearch 数据丢失后，可从 Markdown 重建核心数据。

### 8.2 V2.0：AI 知识问答

#### 版本目标

构建个人 RAG 知识库，支持基于个人知识内容进行可追溯的 AI 问答。

#### 8.2.1 文档切片

- 按标题层级、段落、代码块等结构进行语义切片。
- 保留文档 ID、标题路径、原文位置等追溯信息。
- 控制切片长度与必要的上下文重叠。
- 文档变更后，仅重新处理受影响内容。
- 切片策略应可配置并具备版本标识。

#### 8.2.2 向量化与向量检索

- 对有效切片生成 Embedding。
- 将向量及元数据写入 Milvus。
- 支持按语义相似度召回知识片段。
- 支持分类、标签等元数据过滤。
- Embedding 模型变更后支持索引重建。

#### 8.2.3 AI 知识问答

- 接收自然语言问题。
- 综合关键词检索与向量检索结果生成回答。
- 回答内容应优先基于用户知识库，不确定时明确说明。
- 支持多轮对话，但不得混淆会话上下文与知识库事实。
- 当知识库缺少依据时，避免生成伪造结论。

#### 8.2.4 引用来源

- 回答展示引用文档标题和命中片段。
- 支持跳转至原文对应位置。
- 区分直接引用、综合推断和模型通用知识。
- 保存生成回答所使用的检索结果，便于复现与评估。

#### V2.0 验收标准

- 已同步文档能够完成切片、向量化和检索。
- 典型问题能够召回相关文档片段。
- AI 回答包含可访问、可核对的来源引用。
- 知识库没有相关内容时，系统能够明确提示依据不足。
- 文档修改或删除后，旧向量不会继续作为有效知识被召回。

### 8.3 V3.0：知识图谱平台

#### 版本目标

构建可维护、可探索的技术知识关联网络，帮助用户从单篇文档上升到完整技术体系。

#### 8.3.1 知识关联管理

- 人工创建、编辑和删除知识关系。
- AI 可建议潜在关系，但默认需要用户确认。
- 支持查看关系来源和说明。
- 防止无效自关联与完全重复关系。

#### 8.3.2 知识图谱展示

- 以节点和边展示知识实体及关系。
- 支持缩放、拖动、筛选和节点展开。
- 点击节点可查看详情并跳转至关联文档。
- 支持按分类、关系类型和层级控制展示范围。
- 对大规模图谱采用按需加载，避免一次渲染全部节点。

#### 8.3.3 关联知识推荐

- 在文档详情页展示相关知识。
- 推荐结果说明关联原因。
- 综合人工关系、内容相似度和用户行为进行排序。
- 用户可以忽略或纠正不准确的推荐。

#### V3.0 验收标准

- 用户能够创建并维护知识关系。
- 图谱能够清晰展示指定知识的上下游关联。
- 用户可以从图谱节点访问原始文档。
- AI 生成的关系在确认前不会成为可信事实。
- 关联推荐具备可理解的推荐依据。

### 8.4 V4.0：AI 学习助手

#### 版本目标

在个人知识库与知识图谱基础上，提供个性化、可持续的技术学习与成长辅助。

#### 8.4.1 AI 学习路线

- 用户输入学习目标、当前水平和可投入时间。
- 系统结合已有知识生成阶段化学习路线。
- 标识已掌握、待学习和知识库缺失的主题。
- 每个阶段提供目标、资料、练习和完成标准。
- 学习路线可以保存和更新。

#### 8.4.2 AI 知识补全

- 识别知识体系中的缺失主题和薄弱环节。
- 给出补全建议及原因。
- 生成内容与已有内容明确区分。
- AI 生成内容必须经用户确认后才能进入正式知识库。

#### 8.4.3 AI 知识总结

- 对单篇文档、分类或专题进行总结。
- 支持生成要点、速查表和复习提纲。
- 总结结果保留来源引用。
- 文档更新后提示已有总结可能过期。

#### 8.4.4 AI 知识推荐

- 根据当前阅读内容推荐相关知识。
- 根据知识缺口和学习目标推荐下一步内容。
- 允许用户反馈推荐是否有用。
- 避免长期重复推荐相同内容。

#### 8.4.5 Agent 学习助手

- 将复杂学习目标拆分为可执行任务。
- 在用户授权范围内检索知识、生成练习和检查进度。
- 对外部调用和知识写入保持可见、可确认。
- 记录关键执行过程，支持用户审查。

#### V4.0 验收标准

- 用户输入技术主题后可获得结构化、可执行的学习路线。
- 路线能够区分已有知识和待补充知识。
- 总结与推荐可追溯到知识库来源。
- AI 不能在未经确认的情况下直接修改 Markdown 知识源。
- 用户能够查看、调整和停止 Agent 的学习任务。

## 9. 数据一致性与同步规则

### 9.1 数据权威性

| 数据类型 | 权威来源 | 是否可重建 |
| --- | --- | --- |
| 文档正文与知识元数据 | Markdown | 否，必须备份 |
| 文档结构化记录 | MySQL | 是 |
| 全文索引 | ElasticSearch | 是 |
| 文档切片与向量索引 | Milvus | 是 |
| 收藏、问答历史等用户数据 | MySQL | 需单独备份 |

### 9.2 冲突处理

- Markdown 与派生数据冲突时，以 Markdown 为准。
- 平台内对知识元数据的编辑，最终必须安全回写 Markdown 后才算成功。
- 回写失败时，不得只更新数据库造成双重事实来源。
- 同步前后通过文件标识与内容哈希判断变化。
- 并发修改时应检测冲突，禁止静默覆盖源文件。

### 9.3 删除策略

- Markdown 文件删除后，文档应先进入可恢复的软删除状态。
- 对应全文索引和向量索引应及时失效。
- 恢复期结束后再执行物理清理。
- 用户行为和历史引用中的已删除文档应保留必要的缺失标识。

## 10. 非功能需求

### 10.1 性能

- 常规知识列表与详情页应具备流畅的交互体验。
- 搜索应在个人知识库常见规模下快速返回结果。
- 大批量同步应支持分批处理和进度反馈。
- AI 问答应提供处理中状态，并支持超时与失败重试。

具体量化指标应在明确部署环境和预期数据规模后补充。

### 10.2 可靠性

- 同步任务失败可重试，并保留可诊断日志。
- 外部服务不可用时，不影响 Markdown 原始知识的读取。
- 派生存储能够通过明确流程完成全量重建。
- 关键任务应避免重复消费和部分成功导致的数据不一致。

### 10.3 安全与隐私

- Markdown 渲染必须防范脚本注入。
- 密钥和凭证不得写入知识文档、代码仓库或日志。
- 外部 AI 服务调用应明确数据发送范围。
- 日志避免记录完整敏感文档和用户隐私数据。
- 对文件路径、上传内容和外部链接进行安全校验。

### 10.4 可维护性

- 同步、搜索、向量化和 AI 编排应保持清晰边界。
- 数据模型和索引结构应具备版本管理与迁移机制。
- 核心流程具备自动化测试。
- 重要操作具备结构化日志和必要的监控指标。

### 10.5 可迁移性

- 原始知识可直接通过文件系统和 Git 备份。
- 核心数据应支持导入与导出。
- 搜索、向量和模型供应商应通过抽象边界降低替换成本。

## 11. 关键业务指标

产品早期优先关注实际使用价值，不以公开平台的流量指标为目标。可观察：

- 知识文档总量及每周新增量。
- Markdown 同步成功率与平均同步耗时。
- 搜索成功率、无结果率和结果点击率。
- 文档再次访问率和收藏使用率。
- AI 问答引用覆盖率。
- AI 回答获得有效来源支撑的比例。
- 推荐采纳率和错误关联反馈率。
- 学习路线完成率。

## 13. 风险与应对

| 风险 | 影响 | 应对方向 |
| --- | --- | --- |
| Markdown 规范频繁变化 | 同步逻辑复杂、历史文档不兼容 | 尽早定义最小元数据规范并进行版本管理 |
| 多套存储数据不一致 | 搜索或问答返回过时内容 | 保持 Markdown 权威性，建立幂等同步和重建机制 |
| 文档切片质量不足 | RAG 召回与回答质量下降 | 保留结构信息，建立可评估、可版本化的切片策略 |
| AI 生成错误知识 | 污染个人知识库 | 强制来源引用，AI 写入源文件前必须人工确认 |
| 知识图谱关系失控 | 图谱复杂且不可用 | 限制关系类型，区分建议关系与可信关系 |
| 过早设计复杂架构 | 延长交付周期 | 按版本目标建设，优先完成最小可用闭环 |
| 外部模型泄露隐私 | 敏感知识暴露 | 可配置模型服务，明确发送内容并支持本地模型 |

## 14. 待决策事项

以下内容需要在进入开发前或对应版本启动前进一步确定：

- Markdown 知识库目录结构与命名规则。

- 平台内编辑是否纳入 V1.0，以及安全回写 Markdown 的方式。
- V1.0 的后端、前端和任务调度技术选型。
- 同步触发方式：手动、定时、文件监听或 Git 事件。
- 文档删除后的恢复周期。
- 预期文档数量、单篇大小和代码块规模。
- ElasticSearch 的中文与中英文混合分词方案。
- Embedding 模型、向量维度和 Milvus 集合设计。
- RAG 检索评估数据集与质量标准。
- 知识图谱是否使用独立图数据库。
- 本地模型与外部模型的支持边界。
- 单用户产品向多用户扩展的时机。
